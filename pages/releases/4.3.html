A shared effort

Looking back at the amount of blood, sweat, and tears that went into this one, we can almost guarantee that there's something for everyone in here.

Discover new node types, quality of life changes, and of course many bug fixes in the release overview below.

Highlights

Interactive music

Set the scene with new audio resources

The new resources AudioStreamInteractive, AudioStreamPlaylist, and AudioStreamSynchronized, you enable you to create complex, layered music and transitions.

This allows you to build your own dynamic music system, which reacts to the context of your game.

2D physics interpolation

Now that we call smooth

Physics interpolation is used to decouple physics ticks and display frame rates. It creates additional frames inbetween the last physics position and the current one. This reduces jitter, effectively making for a smoother appearance overall.

This is useful for two things:

  1. If you are making a mobile game, a lower tick rate is less taxing on the hardware you develop for, but normally would have meant compromising on smoothness. Not anymore!
  2. Your games will look better on high refresh displays

Overhauled visual shader editor

Declutter your shader nodes

The visual shader editor has been revamped from the ground up.

Clicking on a node now highlights it for better visibility.

Two new node types have been added: a reroute node and a frame node. Neither of them influence your shader code, rather than being useful to organize your workspace.
You can add a reroute node to any existing connection between nodes and move it freely. A frame node is used to bundle related nodes visually.

TileMap layers turn into TileMapLayer nodes

Layers have become more flexible

Before this release, if you wanted to add layers to your TileMap, you had to add them in the inspector. We simplified this by creating a new TileMapLayer node to replace the old way completely.

Aside from being more intuitive to use, this is now more in line with Godot Engine's design philosophy in general.

For your convenience, you can convert your existing TileMap nodes into TileMapLayer nodes with one click directly in the editor. The deprecated node type is still supported for the time being, if you prefer.

Single-threaded Web exports

Back to the basics

Godot 3 supported both single- and multi-threaded Web exports. When we released Godot 4, we assumed that browser technology would soon improve enough to only focus on multi-threading. Since that did not happen, we are adding back the option to export single-threaded applications.

By doing so, we achive better browser compatibility, making it easier and faster to upload your games to the Web platform. This also removes the need for complicated server-side configurations and other limitations users experienced before.

General multi-threading improvements

Worry less about edge cases

Internal engine components have been reworked to improve the performance of multi-threading across the board. By reducing the chance for edge cases causing deadlocks, loading resources is now much more reliable overall. This effect is noticable even during runtime, no matter the complexity of resources and dependencies!

2D

New Parallax2D node available

Add depth to your backgrounds

Parallax is a visual effect used in 2D games to give players the illusion of depth. This is done by moving background elements at different speeds.

While you were able to add a parallax effect to your games already, the previous implementation of ParallaxLayer came with limitations. For instance, overlaying effects as seen in the video here was impossible before.

Once again, we also built in a converter for your existing scenes.

Improved pixel art stability

Moving pixels align again

What was happening: when snapping transformations to surfaces, users reported gaps and misplaced pixels. This was caused by previously unrefined calculations within those transforms. This could only be circumvented by unsnapping, which was not an ideal solution.

This is now fixed, as can be seen in the video below. Focus on the red markings on the left, and compare to the proper transforms on the right. Notice that the objects properly snap this time.

Animation

New SkeletonModifier3D node

The AnimationMixer grew new bones

Prior to this release, moving bones via script was tricky. The new SkeletonModifier3D node makes this process both easier to grasp and more flexible overall.

Important to note: this is an abstract node, meaning you cannot add it to your projects directly. Instead you can extend it in your scripts for custom functionality.

New skeletal animation import options

Retargeting was never easier

These new import options assist you in retargeting animation sets, especially those using the .fbx format.

To ensure that you'll be able to fully use all the skeletons in your closet, you can now set rest poses, use your own templates, and more.

Enable keyframe manipulation

A quality of life upgrade

You can finally select, copy, paste, and duplicate keyframes.

C#

New C# bindings to get rebuild notifications

A useful reminder

The inspector of your nodes and resources containing .cs scripts will now display a warning whenever you have made changes in your code but not rebuilt the project yet.

Before this change, there was no visible indication of this discrepancy, leading to unecessary debug sessions. Hopefully this will help avoid confusion and save time!

Script reloading process improved

More robust than ever

In the past, there have been user reports of the editor requiring a restart after a rebuilding C# projects. We found the bug that left the editor in invalid states, making the rebuild process less error-prone overall.

Generics now supported out of the box

Being generic is not a bad thing

Generic classes and methods are a stable in many programming languages. They defer the specification of parameter types until runtime, therefore making your code more reusable, type safe, and efficient.

While generics were supported previously, the editor used to throw exceptions when reloading your code. This is now fixed.

Preventing exports with missing C# files

In order to compile any C# project, the engine requires the .sln and .csproj files. If they were missing during export, you ended up with a broken executable - but without attempting to run it, you would not notice the problem.

To prevent this from happening, the editor now does not allow exporting your project without them anymore and shows you a matching error instead.

Display

New rendering driver supported

Direct3D 12

Until now, in order to support Direct3D 12, you had to include the DXIL.dll library at the root of your project yourself. This file used to be proprietary, which prevented us from bundling it with the Godot Engine. Thanks to Microsoft open-sourcing it last month , this is not a problem anymore, enabling us to support this rendering driver directly now.

Since Direct3D is preferred by Windows, this change improves compatibility with any Microsoft platform. Additionally, with Windows coming to ARM recently, this means you can use the Godot Engine on those devices from now on, as well as export your games for them.

Wayland support for Linux/*BSD

Leading the Wayland

Wayland is a modern display server protocol that can be used on Linux and some BSD-derived operating systems. It aims to be a replacement for the X11 window system protocol and architecture. With built-in Wayland support, we allow developers to access to a much cleaner API and whatever new features they release.

Documentation

Commenting enabled on the online docs

"No comment." no more

Following the lead of other online documentations, we added a comment section to the Godot Docs. Please note that it is intended to be used for general corrections and not individual tech-support; for problems with your project please visit the official forums instead.

To comment on a page, you need to be logged into your GitHub account, this is due to the software we are using and helpful for moderation purposes.

In-editor documentation updated

What a glow-up

Did you know, parts of the online and always up-to-date Godot Docs, are available directly in your editor as an offline copy? Note that its contents are tied to the Godot Engine version you are using and are limited to the Class Reference section.

We gave that documentation copy a needed facelift: it now has syntax highlighting. To improve quality of life even further, we also reproduced the copy button from its online pendant: you can now copy the contents of a code block with one click.

Link to specific class references directly

The Class Reference section of the online docs received a handy new feature: you can get a direct link to whatever method you are currently looking at by clicking the link icon next to its name.

Editor

Native ufbx importer

No file conversion necessary

Worry less when importing your .fbx characters and animations, since the file format is now natively supported. This is done via an internal ufbx importer, to remove the need to download an external converter.

By supporting this industry standard, we hope to improve your 3D workflows. This update also comes with some general improvements in this area of the engine.

FileSystem can be moved to the bottom of screen

To further personalize your editor layout, there is now an in-built option to move your FileSystem docker to the bottom of the screen.

This should give you plenty of extra space for your file-viewing needs, and feel more familiar to some.

Layout and UX improvements for the project manager

The design of the project manager has been revamped. You can now directl access some editor settings from here: language, interface theme, display scale, network mode, and directory naming conventions.

Furthermore, we added an update checker, to inform you if a new Godot version has arrived. For it to function automatically, make sure to enable network mode!

GDExtension

Register runtime classes with GDExtension

This feature is particulary interesting for those of you that use GDExtensions to code game logic.

Before, whenever you wanted to register a GDextension class that is only used during runtime, you had to wrap it in a check to see if you were currently in the editor or not (is_editor_hint()).

How you declare a runtime class exactly depends on the bindings you use. For the officially maintained godot-cpp you use GDREGISTER_RUNTIME_CLASS(ExampleRuntime) in your module initialization (screenshot for reference).

Virtual method registration

Ever wanted to register your own virtual methods like _ready(), _process(), or _draw() for users to use in their code?

Now, scripts can implement virtual functions registered by GDExtension classes.

Generate GDExtension documentation with --doctool

GDScript

Binary tokenization on export

Lean and mean

With this new export option, you can reduce the size of the exported GDScript code. But it also offers a way to obfuscate your source code.

While it doesn't protect your work from being decompiled, it at least hides your source code from being available publicly.

Built-in methods and utility functions usable as Callable

You have an array and you want to print each element without creating a for loop?

By making built-in methods and utility functions usable as Callable makes it possible to call variadic functions such as print() with Callable.callv() easily, without the need of boilerplate code.

You can now call this:
print.callv(my_array)

@export_storage property hint

Addon developers, you can now use @export_storage to store hidden values in a scene.

Useful if you need to store data on a node or a resource that need to be accessed by the addon, but that isn't meant to be edited by the user in the inspector.

@export_custom property hint

You need specific types of export? We got you covered.

You can now use @export_custom to export a value suited to your needs.

It allows you to set a custom hint, hint string, and usage flags for the exported property. Great for highly specific exports and for addon developers.

GDScript autocompletion fixes

Do you need a suggestion?

We improved the autocompletion of several components, including enums, subscripts, and get_node().

As it is a difficult feature to detect regressions, we even started doing some automated testing to specifically test autocompletion results.

GDScript circular dependencies between scenes and scripts

A lot of issues with circular dependencies between scenes and scripts have been fixed!

Not only this makes you less worry about cyclic dependencies, but it makes scenes more resilient and stable.

Navigation

Navigation mesh chunks baking

Split and bake!

If you're working on a project with large game worlds, this update will definitely be interesting to you.

Both NavigationPolygon (2D) and NavigationMesh (3D) resources received new properties to define baking bounds, as well as border size offsets. Combined, the two new properties can be used to bake navigation mesh chunks with perfectly aligned edges that fit the neighbor chunks.

Aligned edges allow efficient processing of any runtime changes of the navigation map. Beforehand, helpers were needed to process edges at high cost and it didn't scale for large game worlds.

Navigation obstacle option to affect baking

Our new cookie cutter

NavigationObstacle2D and NavigationObstacle3D now have two new options to affect navigation mesh baking.

Obstacles now discard other source geometry inside their vertices defined shape. They can be either baked with agent radius offset similar to other geometry, or act more like a stencil and carve the shape into the navigation mesh.

In 3D, obstacles baked this way can be used to remove unwanted navigation meshes inside other geometry, like inside walls or roofs.

Navigation path simplification

With the new path simplification option on NavigationAgent nodes, it is now possible to remove less critical path points.

This is useful to resolve some issues that comes with agent movement in large areas. It also helps agents that use navigation points to steer as it leaves only the critical path points as targets.

The algorithm is also exposed from the NavigationServer where it can be used with any generic Vector2/Vector3 array.

Rendering

Premultiplied alpha blending in 3D shaders

Premultiplied alpha is a new blend mode for 3D materials. This enables you to create better looking flames and fireworks, as the same particle now can have the properties of additive blending (for the flame) and mix blending (for the smoke) blending.

See in the video how the left barrel flames (which uses additive blending) are missing the smoke that is visible in the other barrel flames which use premultiplied alpha blending.

Compositor effects API

Compositor effects let users hook into the built in Godot rendering pipeline to add custom rendering code.

They can be used for everything from advanced post processing effects to completely custom rendering!

--ADDITIONAL DEFINITION BY BASTIAAN--

Compositor effects allow a user to inject additional render passes at predetermined points in the render pipeline. This is an advanced feature that does require intimate knowledge of how rendering works but enables users to efficiently create a variety of special effects. Advanced users of Godot can implement new features that regular users can adapt to their needs.

We're already seeing some great examples from early adaptors who are using this feature to create god rays, implement motion blur or completely change the styling of their game.

RenderingDevice context management refactor

We refactored the management of device contexts in the RenderingDevice to fix many hard to track down bugs.

As a bonus, this lets us create new RenderingDevices whenever we want which paved the way for baking lightmaps in the Compatibility rendering backend!

Depth-based fog

Hide and seek

With depth-based fog, you can now decide when the fog begins and when it is fully opaque.

Compared to exponential (and realistic) fog mode, it may seem less realistic, but you now have the tools to create an atmosphere worthy of old-school horror games.

Compatibility rendering backend features

The Compatibility rendering backend received a lot of attention this release cycle. We now consider it feature complete as it has all the features we intended it to have.

New in this release are MSAA, resolution scaling, Glow, ReflectionProbes, LightmapGI, Adjustments, and Color Correction.

Acyclic rendering graph

More for less

RenderingDevice, the part of the engine that powers the Forward+ and Mobile renderering backends, was improved by the introduction of a directed acyclic render graph.

As newer GPU APIs (Vulkan, D3D12, and Metal) give developers more direct access to graphic cards, the burden for making sure everything works correctly falls on the user. The acyclic graph permits the engine to reorder and optimize commands in order to minimize the API overhead of the graphics API while ensuring that everything happens in the right order.

Without making any changes to your 3D scene, you should expect to see a frame rate improvement of 5% to 15% (and more if you heavily use GPUParticles)!

Web

Web Audio samples

Enhanced audio quality

With great power comes great responsibility when it comes to the Web.

As there was audio issues when running games single-threaded, we updated the engine to support audio samples. This enabled us to leverage the Web Audio API in order to deliver seamlessly low-latency and high-quality audio.

Splash screen is now showing up

Not only the Web progress bar has been reworked and optimized, but the splash screen is now showing up before the game loads!

Don't worry: if no boot splash screen has been set, the classic Godot boot splash will be seen, as it is currently displayed on other platforms.

XR

Composition layers to allow 2D viewports to be displayed at much higher quality

Scene discovery/spatial anchors on Meta headsets

Augmented reality requires you to have knowledge of the world around you so you can interact with it. Meta's new scene discovery capabilities allow you to do just that and off course Godot now has support for this feature. From simply getting a mesh of the world around you, to getting informed of key elements of the room such as where walls are, tables are placed, etc.

Spatial anchors let you further decorate this data by identifying points in your space and having the system remember those locations even after restarting your games. This lets you anchor screens, place objects that will reappear in the same location, etc.

Standardized and enhanced support for hand/finger tracking, body tracking, and face tracking

Meta XR simulator support

One difficulty in developing XR games is having to continuously test your application on a headset, especially when developing for stand alone headsets.

The Meta XR simulator is a free option to run your XR game without using your headset.

We've made it easy to configure the simulator to use it directly from Godot. It works both on Windows and on macOS.

Performance and stability improvements

Various improvements have been made to make XR more stable and perform better. We have improved the timing of tracking data for more stable tracking. The core of the XR implementation is now thread safe allowing you to run rendering on a separate thread.

More control has been added over foveated rendering on the mobile and Forward+ renderer. Glasses based AR and passthrough based AR have been unified to make it easier to create augmented reality games without having to worry about hardware differences.
